Why fixed timers waste your time

A conventional traffic light runs a clock: 30 seconds of green this way, 30 seconds that way, all day. It cannot tell whether a direction is empty or jammed, so it holds an empty road green while cars pile up on the cross street. The cost shows up in three numbers that SmartSignal tracks every second of every simulation.

Waiting time

Seconds each driver spends fully stopped. The headline number, and the one the agent is rewarded for cutting.

Queue length

How many cars are halted at the line. Long queues spill back and block upstream junctions.

Throughput

Cars that complete their trip. A good controller clears more cars with less waiting and lower emissions.

How the agent works

SmartSignal frames signal control as a reinforcement-learning loop. Five times a minute the agent looks at the junction, chooses which movement to serve next, and gets scored on whether traffic improved. Over millions of these steps it learns a control policy without ever being told the rules.

Two scenarios to learn on

The demo above runs both. Each uses seeded, time-varying demand so every comparison is reproducible.

Single intersection

4 approaches · 3 lanes each · protected left turns

A four-way junction with dedicated left-turn lanes and four green phases. This is where the agent has to beat a fixed timer, SUMO's actuated control, and max-pressure on its own.

Three demand profiles: calm off-peak, a north-south commuter rush, and a variable hour that ramps from quiet to peak and back, the case where a fixed timer hurts most.

2×2 grid

4 junctions · one shared policy · corridor + balanced demand

Four junctions, each controlled by the same trained policy with added awareness of its neighbours' queues. Coordination has to emerge on its own, no hand-coded timing offsets.

A corridor demand profile sends heavy flow straight through, so the payoff is a green wave: cars cross both junctions with roughly one stop instead of stopping at each.

The four controllers

Beating a fixed timer is easy. The honest comparison is against the adaptive controllers a real traffic engineer would reach for. All four are benchmarked on identical traffic.

Fixed timer baseline

Runs a fixed cycle: each phase gets a set green duration regardless of traffic. The status quo on most streets, and the floor everything else must clear.

Actuated classical

SUMO's built-in gap-based control. It extends a green phase while cars keep arriving and switches when the flow breaks. Genuinely adaptive and widely deployed.

Max-pressure classical

A research-grade rule that serves whichever phase has the largest imbalance between upstream and downstream queues. A strong, principled adaptive baseline.

PPO agent SmartSignal

The learned policy. A neural network trained with Proximal Policy Optimization to choose phases from live conditions. No traffic rules are programmed in; it discovers them.

What the agent sees, does, and learns

The whole policy is defined by three things: its observation, its action space, and its reward. Safety constraints sit between the agent and the road so it can never produce an illegal signal.

Observation what it sees

For every incoming lane: queue density, vehicle density, and normalized waiting time. Plus a one-hot of the active green phase and how long it has been running. All values scaled to [0, 1].

Action what it does

Once every 5 s it picks the index of the next green phase to serve. The environment inserts the mandatory 3 s yellow, holds a minimum 10 s green, and force-rotates after 60 s so no approach is ever starved.

Reward what it learns

The reduction in total accumulated waiting time on incoming lanes since the last decision. Positive when the agent cleared cars, negative when waiting grew. Dense, stable, and tied directly to the headline metric.

Algorithm how it trains

Proximal Policy Optimization (Stable-Baselines3) with a small [128, 128] MLP. About 500k steps across 8 parallel simulations, randomizing the demand profile so the policy generalizes. Roughly 35 minutes on a laptop CPU.

Run it yourself

Everything is reproducible from the command line. The dashboard you are looking at is the last step.

# 1. install dependencies (SUMO ships as a pip wheel)
pip install -r requirements.txt

# 2. generate seeded demand and benchmark the classical controllers
python -m smartsignal.demand.generate_routes
python -m smartsignal.evaluation.run_eval --controllers fixed actuated maxpressure

# 3. train the agent (~35 min on CPU) and evaluate it
python -m smartsignal.training.train_ppo
python -m smartsignal.evaluation.run_eval --controllers rl --model models/ppo_single.zip

# 4. launch this dashboard
uvicorn dashboard.server:app
